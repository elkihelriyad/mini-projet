\section{Problème à traiter}
% ============================================================

\subsection{Contexte}

À l'issue du cycle préparatoire de l'École Nationale des Sciences Appliquées (ENSA) de Safi,
les étudiants doivent choisir l'une des six filières d'ingénierie proposées par l'établissement :
GIIA (Génie Informatique et Intelligence Artificielle), GTR (Génie des Télécommunications et Réseaux),
GINDUS (Génie Industriel), GMSI (Génie Mécatronique et Systèmes Intelligents), GPMA (Génie des Procédés
et Matériaux Avancés) et GATE (Génie Aéronautique et Technologies de l'Espace).

Cette décision est déterminante pour la trajectoire professionnelle de l'étudiant. Or, en l'absence
d'un outil numérique centralisé, les informations sur chaque filière sont dispersées, incomplètes
ou peu accessibles, ce qui contraint les étudiants à s'orienter sans données suffisantes.

\subsection{Problème identifié}

Le problème central est l'\textbf{asymétrie d'information} : l'étudiant ne dispose pas d'un moyen
structuré pour comparer les filières, comprendre leur contenu pédagogique, leurs débouchés
professionnels ou évaluer leur adéquation avec son profil personnel. Cette lacune peut mener
à des orientations inadaptées, source d'insatisfaction académique et d'abandon.

\subsection{Solution proposée}

Le projet \textbf{ENSA Guide} répond à ce besoin en construisant une plateforme web proposant :
\begin{itemize}
    \item Des fiches complètes pour chaque filière (description, programme sur 3 ans, débouchés) ;
    \item Un \textbf{test d'orientation} basé sur un système de scoring dimensionnel adaptatif ;
    \item Un \textbf{chatbot conversationnel} intégré, connecté au modèle Mistral AI ;
    \item Un \textbf{espace personnel} permettant à l'étudiant authentifié de consulter son historique
          de résultats ;
    \item Une \textbf{interface d'administration} pour gérer les comptes et analyser les données.
\end{itemize}

\section{Méthodologie}
% ============================================================

\subsection{Architecture générale}

Le projet est construit selon le patron \textbf{Modèle--Vue--Contrôleur (MVC)}, réparti en trois
couches distinctes :

\begin{itemize}
    \item \textbf{Modèle} : base de données relationnelle SQLite3 stockant les
          spécialités, les questions, les utilisateurs et les résultats de test.
    \item \textbf{Vue} : templates HTML rendus côté serveur via le moteur Jinja2,
          accompagnés de feuilles de style CSS et de scripts JavaScript.
    \item \textbf{Contrôleur} : serveur Flask (Python) organisé en Blueprints,
          gérant le routage, l'authentification, le calcul des scores et les API REST.
\end{itemize}

\subsection{Organisation du code}

Le projet est structuré selon les répertoires suivants :

\begin{lstlisting}[language=bash, caption={Arborescence du projet}]
mini-projet/
|-- backend/
|   |-- app.py               # Point d'entree Flask
|   |-- models/
|   |   `-- database.py      # Connexion et initialisation SQLite
|   |-- routes/
|   |   |-- main_routes.py   # Routes principales (pages HTML)
|   |   |-- api_routes.py    # API REST (test, chat, admin)
|   |   `-- admin_routes.py  # Routes administration
|   |-- services/
|   |   `-- ai_service.py    # Integration Mistral AI
|   `-- utils/
|       |-- helpers.py       # Decorateurs et filtres

    |-- css/                 # Feuilles de style CSS
    `-- js/                  # Scripts JavaScript
\end{lstlisting}

\subsection{Flux de données}

Le cycle de vie d'une requête de l'utilisateur est le suivant :
\begin{enumerate}
    \item L'utilisateur envoie une requête HTTP depuis le navigateur.
    \item Flask l'intercepte via le Blueprint correspondant, vérifie les droits
          de session si nécessaire.
    \item Le contrôleur exécute une requête SQL sur la base SQLite via \texttt{get\_db\_connection()}.
    \item Les données sont injectées dans un template Jinja2 (rendu HTML côté serveur)
          ou retournées au format JSON (API REST).
    \item La réponse est transmise au navigateur, qui l'affiche ou la traite via JavaScript.
\end{enumerate}

\section{Outils et technologies}
% ============================================================

\subsection{Backend : Python et Flask}

Flask est utilisé comme micro-framework web. Son mécanisme de \textit{Blueprints} permet
de découper logiquement le code en modules indépendants (\texttt{main\_routes}, \texttt{api\_routes},
\texttt{admin\_routes}). La bibliothèque \textbf{NumPy} est employée pour les calculs vectoriels
du test d'orientation (normalisation par z-scores, softmax). La sécurité des mots de passe est
assurée par \texttt{werkzeug.security} avec hachage \textit{scrypt}.

\subsection{Base de données : SQLite3}

SQLite3 est retenu pour sa légèreté et son intégration native en Python, sans serveur externe
à configurer. Le fichier \texttt{ensa\_orientation.db} est généré automatiquement au démarrage
si absent, à partir du schéma (\texttt{schema.sql}) et des données initiales (\texttt{data.sql}).

\subsection{Frontend : HTML5, CSS3 et JavaScript}

L'interface est développée en HTML/CSS/JavaScript sans framework tiers. Le moteur de templates
Jinja2 permet le rendu dynamique des pages côté serveur. L'API \texttt{Fetch} JavaScript est
utilisée pour les interactions asynchrones (soumission du test, chatbot).

\subsection{Intelligence artificielle : Mistral AI}

Le chatbot est connecté à l'API Mistral AI via un service dédié (\texttt{ai\_service.py}).
En cas d'indisponibilité de l'API, un message de repli est retourné à l'utilisateur.

\section{Implémentation}
% ============================================================

\subsection{Schéma de la base de données}

La base de données comporte quatre tables principales, définies dans \texttt{schema.sql} :

\begin{lstlisting}[language=SQL, caption={Extrait du schéma SQL (schema.sql)}]
CREATE TABLE specialties (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    code        TEXT NOT NULL UNIQUE,
    name        TEXT NOT NULL,
    description TEXT NOT NULL,
    career_paths TEXT,
    subjects    TEXT,
    additional_info TEXT   -- JSON : programme annee par annee
);

CREATE TABLE questions (
    id       INTEGER PRIMARY KEY AUTOINCREMENT,
    texte    TEXT NOT NULL,
    texte_en TEXT,
    texte_ar TEXT,
    dimensions TEXT NOT NULL  -- ex. 'D2,D3'
);

CREATE TABLE users (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    email          TEXT UNIQUE NOT NULL,
    access_code    TEXT NOT NULL,   -- hache via scrypt
    nom_complet    TEXT,
    role           TEXT,
    is_active      INTEGER DEFAULT 1,
    last_result_json TEXT,
    profile_type   TEXT,
    date_test      DATETIME
);

CREATE TABLE results (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id      INTEGER,
    date_test    DATETIME,
    result_json  TEXT,
    profile_type TEXT,
    duration     INTEGER,
    FOREIGN KEY(user_id) REFERENCES users(id)
);
\end{lstlisting}

Le champ \texttt{additional\_info} de la table \texttt{specialties} contient un objet JSON
décrivant le programme pédagogique complet de la filière sur trois ans (six semestres),
avec la liste des modules par semestre.

\subsection{Initialisation de la base de données}

L'initialisation est déclenchée au démarrage du serveur via \texttt{init\_db()}.
Si le fichier SQLite est absent, le schéma et les données sont chargés automatiquement :

\begin{lstlisting}[language=Python, caption={Initialisation de la base (database.py)}]
import sqlite3
import os

DB_PATH = os.path.join(
    os.path.dirname(os.path.dirname(os.path.dirname(__file__))),
    'database', 'ensa_orientation.db'
)

def get_db_connection():
    conn = sqlite3.connect(DB_PATH)
    conn.row_factory = sqlite3.Row
    return conn

def init_db():
    if not os.path.exists(DB_PATH):
        conn = get_db_connection()
        schema_path = os.path.join(..., 'database', 'schema.sql')
        with open(schema_path, 'r', encoding='utf-8') as f:
            conn.executescript(f.read())
        data_path = os.path.join(..., 'database', 'data.sql')
        if os.path.exists(data_path):
            with open(data_path, 'r', encoding='utf-8') as f:
                conn.executescript(f.read())
        conn.close()
\end{lstlisting}

\noindent\textbf{Remarque :} ce mécanisme crée un problème connu~: si la base de données existe déjà,
les modifications apportées à \texttt{data.sql} ne sont \emph{pas} appliquées automatiquement.
L'utilisateur doit supprimer manuellement le fichier \texttt{.db} pour forcer la réinitialisation.

\subsection{Rendu dynamique des fiches de spécialité}

La route \texttt{/filiere/<code\_filiere>} récupère la spécialité correspondante en base et
effectue un mappage vers les noms de champs attendus par le template Jinja2 :

\begin{lstlisting}[language=Python, caption={Route de rendu des fiches (main\_routes.py)}]
@main_bp.route('/filiere/<code_filiere>')
def filiere_detail(code_filiere):
    conn = get_db_connection()
    filiere = conn.execute(
        'SELECT * FROM specialties WHERE code = ?',
        (code_filiere,)
    ).fetchone()
    conn.close()

    if filiere:
        mapped_filiere = {
            'code':         filiere['code'],
            'nom':          filiere['name'],
            'description':  filiere['description'],
            'debouches':    filiere['career_paths'],
            'mots_cles':    filiere['subjects'],
            'formation_json': filiere['additional_info']
        }
        return render_template('filiere_detail.html',
                               filiere=mapped_filiere)
    return "Filiere non trouvee", 404
\end{lstlisting}

\subsection{Calcul du score d'orientation}

Le test soumet les réponses en format JSON à la route \texttt{/api/submit-test}.
L'algorithme de scoring se déroule en quatre étapes :

\begin{enumerate}
    \item \textbf{Z-normalisation des réponses} : chaque réponse brute est normalisée
          par rapport à la moyenne et à l'écart-type des réponses de l'utilisateur,
          afin de neutraliser les biais de tendance centrale.
    \item \textbf{Calcul des scores dimensionnels} : les questions sont associées à
          des dimensions (D1 à D8, représentant des aptitudes telles que la logique,
          l'informatique, les réseaux, etc.). Le score de chaque dimension est la
          moyenne des z-scores des questions qui lui correspondent.
    \item \textbf{Pondération par filière} : chaque filière est définie par un vecteur
          de poids sur les dimensions, configuré dans \texttt{config.py}. Le score brut
          d'une filière est le produit scalaire de ce vecteur et des scores dimensionnels.
    \item \textbf{Softmax} : les scores bruts sont convertis en probabilités de
          compatibilité (0--100\%) via la fonction softmax, ce qui garantit que leur
          somme est toujours égale à 100\%.
\end{enumerate}

La formule appliquée est :
\begin{equation}
    P(f) = \frac{e^{S(f)}}{\displaystyle\sum_{k} e^{S(k)}} \times 100
    \quad \text{avec} \quad
    S(f) = \sum_{d \in D} w_{f,d} \cdot \bar{z}_d
\end{equation}

où $w_{f,d}$ est le poids de la filière $f$ sur la dimension $d$ et $\bar{z}_d$ est le
score z-normalisé moyen de la dimension $d$.

\begin{lstlisting}[language=Python, caption={Calcul softmax des scores (api\_routes.py)}]
# Ponderation par filiere
filiere_scores = {}
for f_code, weights in WEIGHTS.items():
    score = sum(weights.get(d, 0) * dim_scores[d]
                for d in DIMENSIONS)
    filiere_scores[f_code] = score

# Softmax -> probabilites de compatibilite
exp_scores = {f: np.exp(s) for f, s in filiere_scores.items()}
sum_exp = sum(exp_scores.values())
probabilities = {
    f: (exp_scores[f] / sum_exp) * 100
    for f in exp_scores
}
\end{lstlisting}

\section{Interface utilisateur}
% ============================================================

\subsection{Structure de la navigation}

L'application est composée des pages suivantes :
\begin{itemize}
    \item \textbf{Accueil (\texttt{/})} : section \textit{hero} avec statistiques, puis
          grille des six filières (cartes cliquables).
    \item \textbf{Fiche filière (\texttt{/filiere/<code>})} : description complète,
          programme par année et semestre, débouchés professionnels et mots-clés.
    \item \textbf{Test d'orientation (\texttt{/test-orientation})} : questionnaire
          paginé, barre de progression, rendu des résultats.
    \item \textbf{Tableau de bord (\texttt{/dashboard})} : historique des tests,
          profil calculé sur l'ensemble des résultats passés.
    \item \textbf{Administration (\texttt{/admin})} : gestion des étudiants, activation/
          désactivation des comptes, visualisation des statistiques globales.
\end{itemize}

\subsection{Expérience utilisateur (UX)}

Les choix de design visent à réduire la friction d'usage :
\begin{itemize}
    \item La page d'accueil présente immédiatement les six filières en grille visuelle,
          sans nécessiter de navigation.
    \item Le test d'orientation est présenté question par question avec une barre
          de progression, afin de ne pas décourager l'utilisateur.
    \item Les résultats sont affichés sous forme de pourcentages de compatibilité
          classés par ordre décroissant, accompagnés d'un profil textuel
          (Spécialiste, Hybride ou Équilibré).
    \item Un widget de chatbot flottant est accessible sur toutes les pages,
          permettant de poser des questions contextuelles sans quitter la page en cours.
\end{itemize}

\subsection{Limitations et problèmes rencontrés}

Plusieurs dysfonctionnements ont été identifiés lors des phases de test :

\begin{itemize}
    \item \textbf{CSS non chargé sur certaines routes} : Flask est configuré avec
          \texttt{static\_url\_path='/static'}, mais certains templates utilisent
          des chemins relatifs pour les feuilles de style. Sur des routes imbriquées
          (ex. \texttt{/filiere/GIIA}), le navigateur interprète ces chemins
          différemment, provoquant l'échec du chargement du CSS et la disparition
          totale de la mise en page.
    \item \textbf{Synchronisation SQL non automatique} : Le mécanisme \texttt{init\_db()}
          ne s'exécute que si le fichier \texttt{.db} est absent. Toute modification
          dans \texttt{data.sql} reste donc invisible tant que la base n'est pas
          supprimée manuellement et régénérée.
    \item \textbf{Responsivité partielle} : La mise en page fonctionne correctement
          sur bureau (grille à trois colonnes). Sur mobile, certaines pages de filière
          présentent des débordements de texte et des éléments non réorganisés,
          faute de règles \texttt{@media} suffisamment granulaires.
\end{itemize}

\section{Conclusion}
% ============================================================

Le projet \textbf{ENSA Guide} constitue une réponse fonctionnelle et technique au problème
d'orientation des étudiants en fin de cycle préparatoire à l'ENSA Safi. L'application
couvre l'ensemble du besoin initial : consultation des filières, test d'orientation personnalisé,
chatbot conversationnel et gestion des comptes.

Le choix de la pile technologique (Python/Flask, SQLite, HTML/CSS vanilla) s'est révélé adapté
à un projet de cette envergure~: légèreté du déploiement, absence de dépendances lourdes, et
structure modulaire facilitant la maintenance.

Les principales faiblesses identifiées --- synchronisation SQL, chargement CSS conditionnel,
et responsivité partielle --- relèvent de problèmes d'intégration et de configuration, et non
d'une déficience architecturale. Elles sont donc corrigeables sans refonte majeure du projet.

\section{Perspectives et améliorations}
% ============================================================

\subsection{Corrections techniques prioritaires}

\begin{enumerate}
    \item \textbf{Chargement CSS robuste} : Remplacer tous les chemins relatifs
          des feuilles de style par des appels à \texttt{url\_for('static', filename=...)},
          garantissant une URL absolue indépendante de la route active.

    \item \textbf{Synchronisation de la base de données} : Implémenter un script de
          migration (\texttt{migrate.py}) exécuté séparément, permettant d'appliquer
          des modifications SQL sans supprimer la base existante. Alternativement,
          utiliser Flask-Migrate ou un système de versionnement des schémas.

    \item \textbf{Responsivité complète} : Refactoriser les feuilles de style en
          adoptant une approche \textit{mobile-first}, avec des règles \texttt{@media}
          systématiques pour les grilles et les sections de contenu des fiches de filière.
\end{enumerate}

\subsection{Évolutions fonctionnelles}

\begin{enumerate}
    \item \textbf{Personnalisation du chatbot} : Enrichir le contexte transmis à
          Mistral AI avec les données de la base (descriptions des filières, questions
          fréquentes), afin d'obtenir des réponses plus précises et contextualisées.

    \item \textbf{Tableau de bord administratif enrichi} : Ajouter des graphiques
          de répartition des profils d'orientation pour aider l'administration à
          anticiper les flux vers chaque département.

    \item \textbf{Support multilingue} : Le système de questions dispose déjà de
          champs \texttt{texte\_en} (anglais) et \texttt{texte\_ar} (arabe).
          L'intégration d'un sélecteur de langue dans l'interface permettrait
          d'étendre l'accessibilité de la plateforme.

    \item \textbf{Déploiement en production} : Migrer de SQLite vers PostgreSQL
          pour un déploiement multi-utilisateurs, et conteneuriser l'application
          avec Docker pour simplifier les mises à jour.
\end{enumerate}
